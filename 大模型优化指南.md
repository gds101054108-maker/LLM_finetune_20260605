## 1. 前置知识：大模型训练全流程核心概念说明
基于LlamaFactory框架对大模型训练各阶段的官方定义，梳理核心概念如下：

### 1.1. 预训练（PT, Pre-Training）
- 定义：大模型从0到1的初始训练阶段，使用超大规模通用文本语料（通常是万亿级token），让模型学习通用语言知识、逻辑推理能力、世界常识
- 特点：训练成本极高，需要几十到上百张A100级显卡训练数周/数月，普通开发者几乎不会从零开始做预训练，都是直接用开源/商用的预训练好的基座模型
- LlamaFactory支持：支持基座模型继续预训练，可在通用预训练的基础上补充特定领域的大规模语料，提升模型在垂直领域的基础能力

### 1.2. 继续预训练（CPT, Continue Pre-Training / Domain Pre-Training）
- 定义：在已经预训练好的基座模型基础上，使用特定垂直领域的大规模无标注语料继续训练，让模型学习该领域的专业知识、术语、行业逻辑
- 适用场景：当你需要模型适配专业领域（如法律/医疗/金融），且有几十万条以上的领域无标注语料时使用，是提升垂直领域模型基础能力的最优方案
- LlamaFactory支持：内置继续预训练阶段配置，支持lora/freeze/全参数三种继续预训练模式，显存占用和SFT基本一致

### 1.3 监督微调（SFT, Supervised Fine-Tuning）

#### 定义
监督微调是业务场景最常用的大模型微调方式，通过标注好的「输入-输出」配对样本对预训练基座模型进行定向训练，让模型快速学会遵循特定指令、完成定制化任务（如分类、生成、摘要、对话、推理等），无需从零开始训练模型，落地成本低、见效快。

#### 三种微调模式对比（LlamaFactory原生支持）
| 微调模式 | 核心原理 | 显存占用（单卡batchsize=1） | 训练速度 | 效果上限 | 适用场景 |
|:---------|:---------|:---------------------------|:---------|:---------|:---------|
| **LoRA微调** | 不修改模型原有参数，仅在线性层插入小参数低秩矩阵进行训练 | 0.8B模型：4GB<br>7B模型：8GB | 极快 | 略低于全参数微调 | 快速验证方案、小数据集微调、硬件资源有限场景 |
| **Freeze微调** | 冻结模型底层90%以上参数，仅训练顶部2-8层输出层参数 | 0.8B模型：3GB<br>7B模型：6GB | 快 | 接近全参数微调 | 中小数据集、兼顾效果和训练稳定性的业务落地场景 |
| **全参数微调** | 全量更新模型所有参数 | 0.8B模型：12GB以上<br>7B模型：24GB以上 | 较慢 | 最高 | 百万级以上大数据集、追求最优效果、硬件资源充足的生产场景 |

#### 模式选择指南
1. 首次尝试验证效果：优先选择LoRA微调，训练速度快、资源成本低，1小时内即可得到验证结果
2. 中小规模业务落地：推荐选择Freeze微调，效果接近全参数微调，训练稳定性更高
3. 大规模生产上线：硬件资源充足情况下选择全参数微调，可获得最优的最终效果

### 1.4. 对齐/强化学习阶段
在SFT微调完成后，让模型输出更贴合人类偏好、更符合业务要求的训练阶段，LlamaFactory原生支持以下主流对齐算法：
- **PPO（Proximal Policy Optimization，近端策略优化）**：传统强化学习对齐算法，需要训练奖励模型（Reward Model），通过多轮交互让模型输出更符合人类偏好，训练流程复杂，显存占用高，容易不稳定
- **DPO（Direct Preference Optimization，直接偏好优化）**：当前最主流的对齐算法，不需要单独训练奖励模型，直接使用「偏好对」样本（更好/更差的输出配对）训练，流程简单，训练稳定，显存占用低，效果接近PPO
- **KTO（Knowledge Preference Optimization，知识偏好优化）**：在DPO的基础上优化，同时支持偏好样本和普通SFT样本混合训练，不需要额外构造偏好对，更适合领域微调场景，能同时提升模型知识能力和对齐效果
- **GRPO（Group Relative Policy Optimization，分组相对策略优化）**：PPO的改进版本，不需要训练价值网络，训练更稳定，显存占用更低，比DPO更适合复杂的多轮对话对齐场景，对齐效果更好

### 1.5. 模型蒸馏
用大模型（教师模型）的知识指导小模型（学生模型）训练，让小模型获得接近大模型的效果，同时体积更小、推理速度更快，LlamaFactory支持主流蒸馏算法：
- **白盒蒸馏**：可以访问教师模型的权重、隐藏层输出等内部信息，通过对齐教师模型和学生模型的隐藏层输出、logits分布实现蒸馏，效果更好，训练速度快，需要同时加载教师和学生模型训练
- **黑盒蒸馏**：只能访问教师模型的输入输出接口（比如调用API获取结果），无法获取模型内部信息，通过让学生模型学习教师模型的输出结果实现蒸馏，适配性更强，适合无法获取大模型权重的场景，但是效果略低于白盒蒸馏

#### 📌 CUDA版本说明
✅ 推荐版本：12.1 ~ 12.4，优先选择**12.1**，适配最新PyTorch版本和LlamaFactory所有新特性，兼容性最好。

一个完整的N轮对话，会生成N个独立训练样本，每个样本包含**截止到当前轮次的所有历史对话内容**，对应当前轮次的客户输入和意图标签。例如：
- 10轮的完整对话 → 生成10个样本
- 第1个样本：只包含第1轮对话上下文，对应第1轮意图
- 第2个样本：包含前2轮对话上下文，对应第2轮意图
- ...
- 第10个样本：包含全部10轮对话上下文，对应第10轮意图

### 3.2.3 数据转换脚本（convert_data.py 建议根据自己的业务数据使用智能体deepseek、豆包等生成优化，将自己的数据提供一个样例给智能体生成可用脚本）
```python
import json
import random
from collections import defaultdict

# 加载系统提示词，去掉置信度字段说明
with open("intent_recognition_prompt.txt", "r", encoding="utf-8") as f:
    system_prompt = f.read()
# 移除输出要求里的confidence字段
system_prompt = system_prompt.replace(',\n  "confidence": "<置信度，0-1之间的小数，保留两位小数>"', '')
# 提取任务指令部分（去掉对话上下文占位符之后的内容）
task_instruction = system_prompt.split("## 对话上下文")[0].strip()

def convert_to_alpaca(raw_data_path, output_path):
    with open(raw_data_path, "r", encoding="utf-8") as f:
        raw_data = json.load(f)
    
    alpaca_data = []
    # 处理每个对话案例
    for case in raw_data["test_cases"]:
        case_id = case["case_id"]
        turns = case["dialogue"]
        # 维护当前对话上下文，逐轮累加
        current_context = []
        
        for idx, turn in enumerate(turns):
            current_round = idx + 1
            customer_input = turn["customer_input"]
            true_intent = turn["intent"]
            candidate_responses = turn["candidate_responses"]
            
            # 构造截止到当前轮的对话上下文：历史对话 + 当前客户输入
            dialogue_context_for_sample = current_context.copy()
            dialogue_context_for_sample.append({
                "role": "customer",
                "content": customer_input
            })
            
            # 拼接对话上下文字符串
            dialogue_context_str = "\n".join([
                f"{item['role']}: {item['content']}" 
                for item in dialogue_context_for_sample
            ])
            
            # 构造input字段：对话上下文 + 当前待识别客户输入
            input_content = f"""## 对话上下文（按顺序排列）
{dialogue_context_str}
## 当前待识别客户输入
轮次：{current_round}
客户输入：{customer_input}
请输出识别结果："""
            
            # 构造output字段（无置信度字段）
            output_content = json.dumps({
                "round": current_round,
                "customer_input": customer_input,
                "predicted_intent": true_intent
            }, ensure_ascii=False)
            
            # 添加到训练样本
            alpaca_data.append({
                "instruction": task_instruction,
                "input": input_content,
                "output": output_content
            })
            
            # 随机选一个候选回复作为AI的回答，加到上下文里供下一轮使用
            selected_response = random.choice(candidate_responses)
            current_context.append({
                "role": "ai",
                "content": selected_response
            })
            current_context.append({
                "role": "customer",
                "content": customer_input
            })
    
    with open(output_path, "w", encoding="utf-8") as f:
        json.dump(alpaca_data, f, ensure_ascii=False, indent=2)
    
    print(f"转换完成：共 {len(raw_data['test_cases'])} 个对话，生成 {len(alpaca_data)} 个训练样本，已保存到 {output_path}")

# 转换训练集和测试集
if __name__ == "__main__":
    #convert_to_alpaca("simulation_train_cases_100_final.json", "intent_train.json")
    convert_to_alpaca("simulation_test_cases.json", "intent_test.json")
```

```python
请输出识别结果："""

            selected_response = random.choice(candidate_responses)
            current_context.append({
                "role": "ai",
                "content": selected_response

            current_context.append({
```

```yaml
# 基础模型配置
model_name_or_path: Qwen/Qwen3.5-0.8B # 基座模型本地路径或HuggingFace模型ID
dataset: intent_recognition_train # 训练集名称，对应dataset_info.json里注册的key
template: qwen3 # 模型对话模板，必须和基座模型要求一致（Qwen3系列模型需要使用qwen3模板）
stage: sft # 训练阶段，监督微调必须指定为sft
do_train: true # 核心参数：开启训练模式，必须显式指定
do_eval: true # 开启验证模式，训练过程中自动在测试集上评估效果
eval_dataset: intent_recognition_test # 验证集名称，对应dataset_info.json里注册的key

# LoRA微调专属配置
finetuning_type: lora # 微调类型：lora/qlora/freeze/full，这里选LoRA轻量微调
lora_rank: 8 # LoRA矩阵秩，越小参数越少，显存占用越低，通常8-64
lora_alpha: 16 # LoRA缩放系数，通常设置为rank的2倍
lora_dropout: 0.05 # LoRA层dropout比例，防止过拟合
lora_target: all # 要插入LoRA的模型层，all表示所有线性层，适配性最好

# 输出配置
output_dir: ./output/qwen3.5_0.8b_intent_lora # 训练结果保存路径
overwrite_output_dir: true # 覆盖已存在的输出目录，方便重复训练

# 训练超参数
per_device_train_batch_size: 4 # 单卡训练批次大小，根据显存调整
per_device_eval_batch_size: 4 # 单卡验证批次大小
gradient_accumulation_steps: 2 # 梯度累积步数，等效扩大批次大小 = 单卡批次*卡数*累积步数
learning_rate: 2e-4 # 学习率，LoRA微调通常用1e-4 ~ 5e-4
num_train_epochs: 3 # 1万样本训练3轮效果更好，避免欠拟合
lr_scheduler_type: cosine # 学习率衰减策略，cosine余弦衰减效果最优
warmup_ratio: 0.05 # 学习率预热比例，前5% steps逐步提升学习率，稳定训练
max_grad_norm: 1.0 # 梯度裁剪，限制最大梯度，缓解梯度抖动，稳定训练
weight_decay: 0.01 # 权重衰减，防止过拟合，进一步稳定训练

# 训练流程配置
logging_steps: 10 # 每10步打印一次训练日志
save_strategy: epoch # 模型保存策略，epoch表示每轮训练结束保存一次
eval_strategy: epoch # 验证策略，必须和save_strategy一致，才能开启load_best_model_at_end
load_best_model_at_end: true # 训练结束后自动加载验证集效果最好的模型
metric_for_best_model: eval_loss # 用验证集损失选择最好的模型
greater_is_better: false # 损失越小模型越好

# 性能优化配置
bf16: true # 开启BF16混合精度训练，显存占用减半，训练速度翻倍，需安培及以上架构显卡支持
cutoff_len: 2048 # 输入文本最大长度，超过部分截断，根据业务文本长度调整
report_to: none # 关闭训练日志上报（如TensorBoard/WandB），不需要可以关闭

# 已配置为直接使用独立测试集作为验证集，无需从训练集切分，验证结果更贴合实际业务效果
# ==================== 核心参数说明 ====================
# 以下为本次训练任务中最关键的几个参数及其详细解释：

# 1. stage: sft
#    - 作用: 指定训练阶段，sft(Supervised Fine-Tuning)表示监督微调
#    - 说明: 意图识别任务属于典型的指令微调场景，必须使用sft模式
#    - 其他选项: pt(预训练继续)、rm(奖励模型)、ppo(强化学习)

# 2. finetuning_type: lora
#    - 作用: 微调方法选择
#    - 说明: LoRA通过在原始模型旁路添加低秩矩阵实现微调，仅训练极少量参数(通常<1%)
#    - 优势: 显存占用低(约6-8GB即可)、训练快、可保留基座模型能力
#    - 参数对应: lora_rank=8 → 可训练参数量约=基座模型参数量*rank/模型维度

# 3. lora_rank & lora_alpha
#    - lora_rank=8: 低秩矩阵的维度，控制可训练参数量
#      * 越大模型容量越强，但显存和过拟合风险增加
#      * 8适合0.8B小模型，若效果不佳可尝试16/32
#    - lora_alpha=16: 缩放系数，实际学习率 = learning_rate * (alpha/rank)
#      * 典型配置: alpha = 2 * rank，保持缩放因子为2
#      * 可独立调节: 增大alpha相当于提高LoRA层的影响力

# 4. learning_rate: 2e-4
#    - 作用: 控制参数更新步长
#    - 说明: LoRA微调建议使用1e-4 ~ 5e-4，比全参微调(5e-6)大10-100倍
#    - 调参建议: 
#      * 数据集小( <1000条) → 用1e-4避免过拟合
#      * 数据集大( >10000条) → 可用3e-4 ~ 5e-4加速收敛
#    - 配合调度: cosine衰减 + warmup_ratio=0.05 可使训练更稳定

# 5. cutoff_len: 2048
#    - 作用: 输入文本的最大处理长度(token数)
#    - 说明: Qwen3.5-0.8B最大支持32768 tokens，但意图识别通常文本较短
#    - 显存影响: 长度翻倍 → 显存占用约翻倍(注意力计算复杂度O(n²))
#    - 建议: 
#      * 短意图(如"查天气") → 512足够
#      * 复杂指令(带上下文示例) → 1024~2048
#      * 长文档分类 → 按需增加，但需考虑显存(2048约占用6-8GB)

# 6. per_device_train_batch_size: 4 + gradient_accumulation_steps: 2
#    - 有效批次大小 = 4 * 2 = 8
#    - 作用: 控制每次参数更新前处理的样本数
#    - 调参逻辑:
#      * 显存允许 → 增大batch_size(如4→8)可加速训练
#      * 显存不足 → 减小batch_size(如4→2)，同时增大gradient_accumulation_steps保持有效批次
#      * 有效批次影响: 越大梯度越稳定，但收敛速度可能变慢
#    - 典型范围: 8-64(取决于模型大小和显存)
```

#### DeepSpeed各Stage（零冗余优化级别）功能说明

DeepSpeed通过零冗余优化（ZeRO, Zero Redundancy Optimizer）分片存储训练状态，大幅降低显存占用，共4个优化级别：

| Stage级别 | 优化原理 | 显存占用降低比例 | 适用场景 |
|:----------|:---------|:----------------|:---------|
| **Stage 0（无优化）** | 不做任何显存优化，等价普通分布式训练 | 0% | 小模型训练、显存充足场景 |
| **Stage 1（优化器状态分片）** | 把优化器状态（如Adam的动量、方差）分散到各个显卡上，每张卡只存自己负责的部分 | ~30% | 中等规模模型、显存略有不足场景 |
| **Stage 2（优化器+梯度分片）[本次配置使用]** | 在Stage1基础上，把梯度也分片存储，仅保留当前卡计算所需梯度 | ~50% | 7B-14B模型训练、单卡显存不足场景，兼顾速度和显存优化效果，是最常用的配置 |
| **Stage 3（模型参数全分片）** | 把模型参数也完全分片存储，每张卡只存部分模型参数，训练时动态加载 | ~70%+ | 70B以上大模型训练、多卡集群训练场景，显存优化效果最好但训练速度略有下降 |

---
说明：LlamaFactory源码的`examples/deepspeed`目录下已经内置了所有Stage的官方优化配置文件，无需手动编写复杂配置，直接引用对应路径即可。

from transformers import AutoTokenizer, AutoModelForCausalLM
import torch
import json
import re

# JSON提取容错函数：兼容模型输出带markdown标记、截断情况
def extract_json(text):
    # 优先匹配被```json包裹的内容
    match = re.search(r'```json\s*(.*?)\s*```', text, re.DOTALL)
    if match:
        json_str = match.group(1).strip()
    else:
        # 没匹配到就找第一个{和最后一个}之间的内容
        start = text.find('{')
        end = text.rfind('}')
        if start != -1 and end != -1:
            json_str = text[start:end+1].strip()
        else:
            return text

    # 自动补全缺失的大括号，解决截断问题
    while json_str.count('{') > json_str.count('}'):
        json_str += '}'

    # 自动补全缺失的双引号
    if json_str.count('"') % 2 != 0:
        if not json_str.endswith('"'):
            json_str += '"'

    return json_str

```python
import requests
import os
import re
# 配置项（仅需修改此处）
API_BASE="" #见内部腾讯文档  
MODEL_NAME="qwen3.6-plus"# 可替换为上面列表中支持的任意模型

def extract_json(text):

    if match:
        json_str = match.group(1).strip()
    else:
        start = text.find('{')
        end = text.rfind('}')

# 构造API请求（保持不变）
def chat_completion(messages):
    headers = {
        "Authorization": f"Bearer {API_KEY}",
        "Content-Type": "application/json"
    }
    payload = {
        "model": MODEL_NAME,
        "messages": messages,

    }
    response = requests.post(f"{API_BASE}/chat/completions", headers=headers, json=payload)
    if response.status_code == 200:
```

## 3.10 Bad Case分析与效果优化提升
如果微调后模型准确率提升不明显、甚至低于基线模型，可按照以下流程分析错误案例（Bad Case），针对性优化训练集和配置，快速提升效果。

### 3.10.1 Bad Case常见错误类型分类
先将`error_cases.json`中的错误案例按原因分类，常见类型如下：

| 错误类型 | 表现特征 |
|----------|----------|
| **标注错误类** | 测试集/训练集本身的意图标签打错，比如客户问保费金额，真实标注意图写成`insurance_expiry_inquiry`，导致评估结果不准或模型学错 |
| **输出格式错误类** | 模型输出不符合JSON格式要求，比如多了说明文字、括号不匹配、引号缺失，或者意图ID拼写错误（如`premium_consult`写成`premium_consalt`），导致解析失败 |
| **上下文理解错误类** | 多轮对话场景下，模型未参考历史上下文，仅根据客户最后一句话识别，比如上下文已经提到续保话题，客户说“多少钱”，模型识别为其他意图而非`premium_consult` |
| **样本覆盖不足类** | 某种意图的训练样本量少，或者特定口语化表述未在训练集中出现，比如客户说“我已经买完别家的了”，训练集中无类似表述，模型无法识别为`refuse_insurance_renewal` |
| **过拟合类** | 训练集准确率接近100%，但测试集准确率远低于训练集，泛化能力差，常见于小数据集全参数微调场景 |
| **欠拟合类** | 训练集和测试集准确率都很低，模型未学会任务规则，常见于训练轮数太少、学习率过低、LoRA rank太小等场景 |

### 3.10.2 Bad Case分析步骤
#### 第一步：排查数据层面问题（优先排查，80%的效果问题都来自数据）
1. 先检查测试集的错误样本标签是否正确，排除标注错误导致的准确率虚低。
2. 检查训练集中是否有对应错误场景的样本：
   - 如果错误类型集中在某几个特定意图，说明这几个意图的训练样本不足、或者样本多样性不够。
   - 如果错误的表述在训练集中完全没有出现，说明样本覆盖不足。
3. 检查训练集是否有噪声：是否存在标签错误的样本，比如把A意图的样本标注为B意图，导致模型学偏。

#### 第二步：排查训练配置问题
1. 检查训练超参数：
   - 学习率是否过大导致模型训崩（训练loss震荡不下降），或者过小导致模型未收敛（训练loss下降慢）。
   - epoch数是否太少导致欠拟合，或者太多导致过拟合。
   - LoRA rank是否太小导致模型容量不足，或者太大导致过拟合。
2. 检查Prompt对齐：推理时使用的系统提示词、输入格式是否和训练时完全一致，Qwen系列模型必须加`system`角色才能保证效果。
3. 检查数据格式：训练集是否符合Alpaca格式要求，是否有字段缺失、格式错误。

#### 第三步：排查模型能力问题
如果数据和配置都没问题，准确率仍达不到预期：
- 小模型（如0.8B）对复杂多轮对话的理解能力有限，可尝试换更大的基座模型（如1.8B/7B）。
- 检查是否训练样本量不足，可通过数据扩增或者补充真实标注样本提升效果。

### 3.10.3 针对性优化方案
#### 1. 数据层面优化（优先级最高）
- **修正标注错误**：清理训练集和测试集的噪声数据，修正错误标签。
- **补充缺失样本**：针对错误率高的意图，补充更多不同表述的训练样本，比如`refuse_insurance_renewal`错误多，就补充“我已经买别家了”“不需要，谢谢”“太贵了不买”等多种口语化表述的样本。
- **数据扩增**：对样本少的意图做同义替换、上下文组合扩增，增加样本多样性，提升泛化能力。
- **平衡样本分布**：如果某类意图样本量远低于其他类，可适当做过采样，或者在训练时增加该类样本的损失权重。

#### 2. 训练配置优化
| 问题类型 | 优化方案 |
|----------|----------|
| 欠拟合（训练/测试准确率都低） | 增大学习率（LoRA微调可从2e-4调整到3e-4~5e-4）、增加训练epoch、增大LoRA rank（从8调整到16/32）、改用Freeze微调模式 |
| 过拟合（训练准确率高，测试准确率低） | 减少训练epoch、增大Dropout、减小学习率、减小LoRA rank、增加训练样本量、改用LoRA/Freeze模式替代全参数微调 |
| 输出格式错误多 | 在系统提示词中强化格式要求，训练集中增加更多格式正确的样本，或者在推理时优化JSON提取容错逻辑 |

#### 3. 推理层面优化
- 优化JSON提取逻辑，增加更多容错处理：自动补全缺失的括号、引号，模糊匹配意图ID（比如模型输出的意图和已有意图相似度超过90%自动匹配）。
- 推理时设置`temperature=0.01`，关闭采样，保证输出稳定，减少随机错误。
- 严格对齐训练和推理的Prompt格式，包括系统提示词、上下文拼接方式、输入结构，保证完全一致。

### 3.10.4 优化效果验证流程
1. 优化完成后重新训练模型，在相同测试集上重新评估。
2. 对比优化前后的准确率变化，重点关注之前错误的Bad Case是否已经被正确识别。
3. 如果仍有错误，重复“分析Bad Case→针对性优化→重新训练测试”的迭代流程，直到达到预期准确率。
